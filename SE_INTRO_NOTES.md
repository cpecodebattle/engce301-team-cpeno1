# บทบาทของวิศวกรซอฟต์แวร์ (Software Engineer Role)
วิศวกรซอฟต์แวร์คือผู้รับผิดชอบในการวางแผน ออกแบบ พัฒนา และดูแลระบบซอฟต์แวร์ให้ทำงานได้ถูกต้องและมีประสิทธิภาพ พวกเขาต้องวิเคราะห์ความต้องการของผู้ใช้ แปลงให้เป็นโครงสร้างระบบ และเขียนโค้ดตามมาตรฐานที่ดี รวมถึงตรวจสอบคุณภาพด้วยการทดสอบในหลายระดับ วิศวกรซอฟต์แวร์ยังต้องแก้ไขปัญหาและบำรุงรักษาระบบหลังใช้งานจริง เพื่อให้ระบบมีความเสถียร ปลอดภัย และรองรับการขยายตัว นอกจากนี้ยังทำงานร่วมกับทีมอื่น เช่น นักออกแบบ นักทดสอบ และผู้เกี่ยวข้อง เพื่อให้ซอฟต์แวร์ตรงตามเป้าหมายของธุรกิจและผู้ใช้ที่สุด
## ระบุตัวอย่างงาน/กิจกรรม
1. วางแผนและออกแบบ
    
        -วิเคราะห์ความต้องการ: รวบรวมและสรุปฟังก์ชันที่ระบบต้องทำ
        -ออกแบบโครงสร้างสร้างแผนภาพสถาปัตยกรรมและออกแบบตารางฐานข้อมูล                           
2. พัฒนาและติดตั้ง
    
        -เขียนโค้ดพัฒนาฟีเจอร์ตามการออกแบบ
        -ควบคุมเวอร์ชันติดตามและรวมโค้ดที่พัฒนา
        -ติดตั้งนำซอฟต์แวร์ขึ้นใช้งานจริง

3. ทดสอบและบำรุงรักษา 

        -ทดสอบตรวจสอบการทำงานของโค้ดและระบบเพื่อหาข้อบกพร่อง
        -บำรุงรักษาแก้ไขปัญหาหลังการใช้งาน และปรับปรุงโค้ดให้ดีขึ้น

# ภาพรวม SDLC (Software Development Life Cycle)
```
+--------------+    +--------+    +----------------+    +---------+    +-----------+    +------------+
| Requirements | -> | Design | -> | Implementation | -> | Testing | -> | Deployment| -> | Maintenance|
+--------------+    +--------+    +----------------+    +---------+    +-----------+    +------------+
```
## Requirements
รวบรวมสิ่งที่ผู้ใช้ต้องการและวิเคราะห์ให้ชัดเจนว่าระบบควรทำงานอย่างไร รวมถึงข้อจำกัดหรือเงื่อนไขต่าง ๆ ที่ต้องคำนึงถึง
หากขั้นตอนนี้คลาดเคลื่อน ตั้งแต่ต้นจนปลายงานจะได้รับผลกระทบ ทำให้ต้องแก้ย้อนหลังและเพิ่มต้นทุนโดยไม่จำเป็น
## Design
แปลงความต้องการให้เป็นแบบร่างโครงสร้างระบบ ตั้งแต่สถาปัตยกรรม ฐานข้อมูล ไปจนถึงส่วนติดต่อผู้ใช้อย่างคร่าว ๆ
การออกแบบที่เป็นระบบช่วยให้การพัฒนาราบรื่น ดูแลแก้ไขง่าย และรองรับการเพิ่มฟีเจอร์ใหม่ได้ในอนาคต
## Implementation
นำแบบที่กำหนดไว้มาเขียนเป็นโค้ดจริง พัฒนาเป็นโมดูลต่าง ๆ ให้ทำงานร่วมกันตามสเปกที่วางไว้
ตรงนี้เป็นจุดที่ไอเดียถูกสร้างเป็นระบบจริง และคุณภาพของโค้ดจะส่งผลต่อความเสถียรและความยุ่งยากในการดูแลระยะยาวอย่างมาก
## Testing
ตรวจสอบคุณภาพซอฟต์แวร์ผ่านการทดสอบหลายระดับ เพื่อค้นหาบั๊กและพิสูจน์ว่าฟังก์ชันต่าง ๆ ทำงานถูกต้อง
ถ้าข้ามขั้นตอนนี้หรือทดสอบไม่ละเอียด ปัญหาจะหลุดไปถึงผู้ใช้ สร้างความเสียหายทั้งด้านภาพลักษณ์และค่าใช้จ่ายในการแก้ไข
## Deployment
นำซอฟต์แวร์ที่ผ่านการทดสอบขึ้นใช้งานจริง พร้อมปรับแต่งค่าระบบและสภาพแวดล้อมให้เหมาะสม
ช่วงนี้มีผลต่อความพร้อมใช้งาน ความปลอดภัย และการเปลี่ยนผ่านจากระบบเดิมไปสู่ระบบใหม่โดยไม่สะดุด
## Maintenance
ติดตามแก้ไขข้อผิดพลาดที่พบหลังเปิดใช้งาน และพัฒนาระบบให้ตอบรับความต้องการที่เปลี่ยนแปลงอยู่เสมอ
เป็นงานที่ต่อเนื่องยาวนาน และมีบทบาทสำคัญต่อคุณภาพการใช้งานในระยะยาวว่าระบบจะยังคงเสถียรและเติบโตได้หรือไม่

# Waterfall vs Agile – ตารางเปรียบเทียบ
| หัวข้อเปรียบเทียบ | Waterfall | Agile (Sprints) |
|--------------------|-----------|------------------|
| ลักษณะการทำงาน | ทำงานเป็นลำดับขั้นตอน | ทำงานเป็นรอบสั้น ๆ สามารถปรับเปลี่ยนได้ทุกเมื่อ |
| ความยืดหยุ่น | ต่ำ เปลี่ยน requirement ยาก | สูง ปรับเปลี่ยนได้ตาม feedback |
| การวางแผน | ต้องละเอียดตั้งแต่ต้น | วางแผนสั้น ๆ เฉพาะ Sprints |
| ความเสี่ยง | สูง เพราะปัญหามักจะเจอตอนท้าย | ต่ำ เพราะทดสอบและปรับทันทีในแต่ละรอบ |

# การประยุกต์ใช้กับ Term Project ของกลุ่มเรา

## 1. ถ้า Term Project ใช้ Waterfall 100% กลุ่มเราคิดว่าจุดเด่น/ข้อจำกัดคืออะไร?
จุดเด่น
    - มีขั้นตอนลำดับการทำงานแบ่งกันชัดเจน
    - วางแผนง่ายและติดตามความคืบหน้าได้เป็นระบบ
    - โฟกัสในการทำงานได้ 100% ไม่ต้องห่วงหน้าพะวงหลัง
    - สามารถคาดการณ์ระยะเวลาหรืองบประมาณได้เพราะกำหนดตั้งแต่ต้น
จุดด้อย
    - ถ้าหากมี requirement เพิ่ม จะกระทบกับโปรเจคทั้งระบบ
    - ทีมงานต้องวิเคราะห์วางแผนอย่างเคร่งครัด ถ้าไม่ละเอียดพอจะก่อให้เกิดปัญหาในขั้นตอนหลัง แก้ไขได้ยาก
    - ไม่เหมาะกับโปรเจคที่ต้องทดสอบ/ปรับปรุงแก้ไขหลาย ๆ รอบ เพราะต้องรอจนกว่าจะถึงขั้นตอนทดสอบ  

## 2. ถ้า Term Project ใช้ Agile (แบ่งเป็น Sprints) จุดเด่น/ข้อจำกัดคืออะไร?
จุดเด่น
    - สามารถปรับเปลีี่ยน requirement ได้ง่าย เพราะวางแผนระยะสั้น
    - หากได้ feedback จากผู้ใช้งานได้เร็ว ก็จะแก้ไขทิศทางตามผู้ใช้ได้ทันท่วงที
    - ส่งงานเป็นส่วนย่อยได้ตลอดเวลา ทำให้รู้ปัญหาได้ไวและแก้ไขได้ทัน
จุดด้อย
    - อาจจะต้องมีการประชุมและวางแผนบ่อยกว่า
    - แผนสามารถเปลี่ยนได้ตลอด ความคืบหน้าอาจไม่แน่นอน
    - ถ้าคนในทีมสื่อสารผิดพลาด งานแต่ละ Sprints จะไม่เสร็จหรือตรงตามเป้าหมาย
    - ถ้าไม่กำหนดขอบเขต หรือขอบเขตไม่ชัดเจน อาจทำให้โปรเจคออกทะเลขึ้นเรื่อย ๆ


## 3. สุดท้าย กลุ่มเราอยากใช้ approach แบบไหน หรือ hybrid แบบใด? เพราะเหตุใด?

กลุ่มเราใช้แบบ Water + Agile Waterfall  เพื่อให้รู้ว่า "เรากำลังจะสร้างตึกอะไร" โครงสร้างและงบประมาณต้องนิ่ง จากนั้นเราใช้ Agile ตามมา เพื่อให้ ตกแต่งตึกนั้นให้ถูกใจลูกค้าที่สุด (ปรับเปลี่ยนรายละเอียดตาม Feedback ได้ตลอด) กลุ่มเรามี แผนงานหลัก ที่ไม่หลุดกรอบง่าย  ตอบโจทย์ผู้ใช้งาน ได้ดีขึ้นเพราะมีการปรับปรุงเป็นระยะ ทำให้ผลิตภัณฑ์สุดท้ายมีโอกาสประสบความสำเร็จสูง
